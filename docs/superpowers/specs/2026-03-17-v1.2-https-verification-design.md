# Design Specification: v1.2 HTTPS Verification System

**Date:** 2026-03-17
**Author:** Claude Code
**Status:** Draft
**Milestone:** v1.2 HTTPS with Automatic Certificates

## 1. Overview

This specification defines a comprehensive automated testing system to validate and complete the v1.2 HTTPS milestone for Portless.NET. The milestone currently shows 18 unverified requirements across phases 15-19, despite significant implementation already existing in the codebase.

### Problem Statement

The archived v1.2 roadmap shows phases 15-19 as "deferred," but code inspection reveals that most features are already implemented:
- HTTPS endpoint with dual ports (1355/1356)
- Certificate monitoring and renewal services
- Integration tests for mixed protocol routing
- CLI commands for certificate management

**The gap is formal verification:** No VERIFICATION.md files exist to validate that requirements are met.

### Solution

Create a 4-layer testing architecture that systematically validates each of the 18 missing requirements through automated tests, with documented evidence in VERIFICATION.md files.

## 2. Architecture

### 4-Layer Testing Structure

```
LAYER 1: Unit Tests          → Phase 15 (HTTPS Endpoint)
   ↓
LAYER 2: Integration Tests   → Phase 16 (Mixed Protocol)
   ↓
LAYER 3: Lifecycle Tests     → Phase 17 (Certificate Lifecycle)
   ↓
LAYER 4: E2E Tests          → Phase 18 (Full Integration)
   ↓
VERIFICATION.md Files       → Phase 19 (Documentation)
```

### Key Principles

1. **Independence:** Each test can run standalone
2. **Reproducibility:** Deterministic results without external dependencies
3. **Speed:** Unit < 100ms, Integration < 2s, E2E < 10s
4. **Evidence:** Every test generates output for VERIFICATION.md

## 4. Testing Strategy by Phase

### Phase 15: HTTPS Endpoint - Unit Tests

**Focus:** Validate Kestrel configuration without starting real server

**Technique:** `IWebHostBuilder` with `TestServer`

**Mocking:** Certificate services return fake certificates

**Requirements:**
- HTTPS-01: Dual endpoints (HTTP 1355, HTTPS 1356)
- HTTPS-02: Wildcard certificate binding
- HTTPS-03: TLS 1.2+ minimum protocol enforcement
- HTTPS-04: Kestrel HTTPS configuration
- HTTPS-05: `--https` flag for `proxy start`
- CLI-05: HTTPS port configuration (PORTLESS_HTTPS_PORT env var)

### Phase 16: Mixed Protocol - Integration Tests

**Focus:** Real YARP with fake backends

**Technique:** `WebApplicationFactory` + `TestServer`

**Setup:** Create fake HTTP and HTTPS backends

**Requirements:**
- MIXED-01: X-Forwarded-Proto header preservation
- MIXED-02: Mixed routing support (HTTP/HTTPS backends)
- MIXED-03: YARP backend SSL validation (self-signed certs)
- MIXED-04: Backend protocol detection
- MIXED-05: Forwarded headers middleware configuration

### Phase 17: Certificate Lifecycle - Time-Mocked Tests

**Focus:** Certificate expiration with time manipulation

**Technique:** `TimeProvider` abstraction for mocking `DateTimeOffset.UtcNow`

**Scenarios:**
- Certificate expires tomorrow → startup warning
- Certificate expired yesterday → startup error
- Certificate valid for 6 months → no warning
- Background service auto-renews

**Requirements:**
- LIFECYCLE-01: Startup expiration check (30-day warning)
- LIFECYCLE-02: Background hosted service (6-hour intervals)
- LIFECYCLE-03: Auto-renewal within 30 days
- LIFECYCLE-04: Certificate metadata storage (cert-info.json)
- LIFECYCLE-05: Manual `portless cert renew` command
- LIFECYCLE-06: Restart requirement documentation
- CLI-03: Renew command implementation
- CLI-06: Renewal notification implementation

### Phase 18: Full Integration - E2E Tests

**Focus:** End-to-end scenarios with real Portless services

**Technique:** Real processes via `Portless.E2ETests` project

**Setup:**
- Start real proxy with `portless proxy start --https`
- Create test backend with `dotnet run`
- Execute real CLI commands

**Requirements:**
- TEST-01: Certificate generation integration tests
- TEST-02: HTTPS endpoint integration tests
- TEST-03: X-Forwarded-Proto preservation tests
- TEST-04: Certificate renewal tests
- TEST-05: Trust status detection tests (Windows)
- TEST-06: Mixed protocol routing tests

### Phase 19: Documentation - Meta-Tests

**Focus:** Validate documentation exists and is correct

**Technique:** File system checks + Markdown parsing

**Requirements:**
- DOCS-01: Certificate management user guide
- DOCS-02: Troubleshooting guide
- DOCS-03: Migration guide from v1.1 to v1.2
- DOCS-04: Platform-specific notes
- DOCS-05: Security considerations

## 6. Verification Workflow

### Step 1: Environment Preparation
```bash
dotnet clean
rm -rf ~/.portless/
dotnet build Portless.slnx
dotnet test Portless.Tests/Portless.Tests.csproj
```

### Step 2: Execute Tests by Phase
```bash
dotnet test --filter "FullyQualifiedName~Phase15"  # Unit
dotnet test --filter "FullyQualifiedName~Phase16"  # Integration
dotnet test --filter "FullyQualifiedName~Phase17"  # Lifecycle
dot test test --filter "FullyQualifiedName~Phase18"  # E2E
dotnet test --filter "FullyQualifiedName~Phase19"  # Documentation
```

### Step 3: Generate Evidence
- Test output captured in TRX format
- Script `generate-verification.ps1` parses results
- Creates VERIFICATION.md with tables

### Step 4: Manual E2E Validation
- Browser verification for HTTPS warnings
- Screenshots saved to `.planning/archived/phases/*/evidence/`
- VERIFICATION.md includes screenshot links

### Step 5: Create VERIFICATION.md

Format:

## 8. Implementation Order

1. **Phase 15** (Unit Tests) - Foundation, no dependencies
2. **Phase 16** (Integration Tests) - Depends on Phase 15 passing

## 9. Risks and Mitigations

### Risk 1: Time-Dependent Tests
**Probability:** High | **Impact:** Medium

Time-based tests can be flaky if not properly mocked.

**Mitigation:** Use `TimeProvider` abstraction, freeze time in tests, avoid `DateTimeOffset.UtcNow` directly.

### Risk 2: E2E Test Environment
**Probability:** Medium | **Impact:** High

E2E tests require clean environment, may fail if ports occupied.

**Mitigation:** Use dynamic port allocation, cleanup in teardown, run in isolated containers if possible.

### Risk 3: Windows-Specific Features
**Probability:** Medium | **Impact:** Medium

Certificate trust installation is Windows-only.

**Mitigation:** Platform-specific test guards `[SupportedOSPlatform("windows")]`, skip tests on other platforms with clear message.

### Risk 4: Certificate State
**Probability:** Low | **Impact:** Low

Tests may generate certificates that persist and affect subsequent runs.

**Mitigation:** Use test-specific certificate directories, cleanup in test teardown, check for stale certificates before tests.
