# Portless.NET v2.0 - Vision & Feature Planning

**Date:** 2026-03-31
**Status:** DRAFT - Pending Review

## Executive Summary

Portless.NET v1.0-1.3 logro feature parity basica con Portless Node.js. v2.0 debe **superarlo** aprovechando las ventajas unicas de .NET 10 + YARP + Kestrel para ofrecer prestaciones que el original no tiene y no puede tener facilmente.

### TIER 1: Developer Experience Completa (Paridad + UX)

#### 1.1 CLI Completa
**Comandos nuevos:**
- `portless get <name>` - Retorna URL para scripts (ej: `BACKEND_URL=$(portless get api)`)
- `portless alias <name> <port>` - Rutas estaticas para Docker/servicios externos
- `portless alias --remove <name>` - Eliminar alias
- `portless hosts sync` - Sincronizar /etc/hosts
- `portless hosts clean` - Limpiar entradas de portless

**Mejoras:**
- Auto-deteccion de nombre desde:
  1. `.csproj` <AssemblyName> o <PackageId>
  2. Directory.Build.props <ProjectName>
  3. git repo root directory
  4. cwd basename
- `--name` override en `portless run`
- Auto-start del proxy si no esta corriendo

**Esfuerzo:** ~1 semana

#### 1.2 Framework Detection Inteligente
**Extension del concept del original pero adaptado a .NET:**

| Framework | Deteccion | Inyeccion |
|-----------|-----------|-----------|
| `dotnet` / ASP.NET | *.csproj con Microsoft.AspNetCore* | `ASPNETCORE_URLS=http://0.0.0.0:{PORT}` |
| `dotnet run` | Proyecto .csproj | `ASPNETCORE_URLS` + `--urls` |
| Vite | vite.config.* | `--port {PORT} --strictPort --host` |
| Next.js | next.config.* | `PORT={PORT}` |
| npm/node | package.json con start script | `PORT={PORT}` |
| Python (Flask/Django/FastAPI) | requirements.txt | `PORT={PORT}` |
| Go | go.mod | `PORT={PORT}` |
| Rust (cargo) | Cargo.toml | `PORT={PORT}` |
| Docker | docker-compose.yml | Publicar puerto del contenedor |

**Ademas:** Placeholder expansion en comandos:
- `{PORT}` -> puerto asignado
- `{HOST}` -> 127.0.0.1
- `{URL}` -> URL publica (ej: https://myapp.localhost)

**Esfuerzo:** ~1 semana

#### 1.3 Paginas de Error Branded
- 404: Lista de rutas activas con links clickeables
- 502: Error contextual (proceso muerto, puerto cerrado, etc.)
- 508: Loop detection con instrucciones de fix
- Soporte dark/light mode
- Design limpio tipo Vercel/Portless pero con branding Portless.NET

**Esfuerzo:** ~3 dias

### TIER 2: Routing Avanzado (Diferenciadores clave)

#### 2.1 Path-Based Routing
**El feature mas pedido en el original (#180) que nunca implementaron.**

```
portless run api dotnet run --path /api
portless run web npm start --path /
```

Resultado:
- `https://myapp.localhost/api/*` -> api (puerto 4001)
- `https://myapp.localhost/*` -> web (puerto 4002)

**Implementacion:** YARP soporta path-based routing nativamente con RouteConfig.Match.Path.

**Esfuerzo:** ~3 dias

#### 2.2 TCP Proxying para Databases
**Segundo feature mas pedido (#181).**

```yaml
# portless.config.yaml (nuevo - ver 2.4)
services:
  postgres:
    type: tcp
    target: localhost:5432
    proxy: postgres.myapp.localhost:15432
```

Permite acceder databases, Redis, etc. via URLs amigables. Ideal para monorepos.

**Implementacion:** Kestrel puede hacer TCP proxying con YARP streams.

**Esfuerzo:** ~1 semana

#### 2.3 Multi-Backend Load Balancing
**El original solo soporta 1 backend por ruta.**

```
portless run api dotnet run --replicas 3
```

YARP round-robin entre 3 instancias del backend. util para:
- Testear load balancing real en local
- Simular produccion con multiples instancias

**Esfuerzo:** ~4 dias

### TIER 3: Configuracion y Observabilidad

#### 2.4 Configuration File (portless.config.yaml)
**El original solo tiene env vars. Nosotros podemos tener declarative config.**

```yaml
# portless.config.yaml
proxy:
  port: 443
  https: true
  
tld: localhost

services:
  api:
    command: dotnet run --project src/Api
    path: /api
    
  web:
    command: npm start
    path: /
    
  postgres:
    type: tcp
    port: 5432
    alias: true
    
  redis:
    type: tcp
    port: 6379
    alias: true

certificates:
  auto-trust: true
  validity: 365  # days
  
environment:
  ASPNETCORE_ENVIRONMENT: Development
  NODE_ENV: development
```

**Comandos:**
- `portless up` - Levanta todo segun config file
- `portless down` - Para todo
- `portless config validate` - Valida el config

**Esfuerzo:** ~1.5 semanas

#### 2.5 Metrics & Observability
**El original no tiene nada de esto.**

- Endpoint `/metrics` en formato Prometheus
- Contadores: requests/sec por ruta, latencia, errores, bytes
- `portless status` con dashboard en terminal (Spectre.Console)
- Health checks de backends con auto-disable
- Structured logging a archivo

**Esfuerzo:** ~1 semana

#### 2.6 Health Checks
YARP soporta health checks nativos. Si un backend esta down:
- Marcar como unhealthy
- Retornar 502 con info util
- Auto-reintentar periodicamente
- `portless status` muestra salud de cada servicio

**Esfuerzo:** ~3 dias

### TIER 4: Deploy & Distribucion

#### 3.1 Native AOT Single Binary
**Ventaja unica vs Node.js:**

```bash
# Un solo archivo, sin runtime, startup instantaneo
dotnet publish Portless.Cli -c Release -r win-x64 -p:PublishAot=true
# Output: portless.exe (~15MB, self-contained)
```

- Windows: `portless.exe`
- Linux: `portless` (single binary)
- macOS: `portless` (Universal binary arm64+x64)

**Esfuerzo:** ~3 dias (mayormente resolver trimming warnings)

#### 3.2 NuGet Package como dotnet tool
```bash
dotnet tool install --global Portless.DotNet
portless proxy start
```

Ya tenemos el manifest. Solo falta pulir y publicar.

**Esfuerzo:** ~2 dias

#### 3.3 Shell Completion
```bash
# Bash
eval "$(portless completions bash)"
# Zsh
portless completions zsh > ~/.zfunc/_portless
# PowerShell
portless completions powershell | Out-String | Invoke-Expression
```

Spectre.Console tiene soporte parcial. Lo extendemos.

**Esfuerzo:** ~3 dias

## Ventajas Competitivas de Portless.NET vs Portless Node.js

| Aspecto | Portless (Node.js) | Portless.NET |
|---------|-------------------|--------------|
| Runtime | Requiere Node.js 20+ | Single binary (AOT) |
| Crypto | Depende de openssl CLI | Crypto .NET integrado |
| Proxy | http2 custom (439 lines) | YARP (production-grade) |
| Performance | Single-threaded | Multi-threaded Kestrel |
| HTTP/3 | No | Si (QUIC nativo) |
| gRPC | No | Si (YARP) |
| Path routing | No (feature request) | Si (v2.0) |
| TCP proxy | No (feature request) | Si (v2.0) |
| Load balance | No | Si (YARP, v2.0) |
| Config file | No (solo env vars) | Si (YAML, v2.0) |
| Metrics | No | Si (Prometheus, v2.0) |
| Health checks | No | Si (v2.0) |
| Windows | Parcial (requiere OpenSSL) | Nativo (crypto integrado) |
| E2E tests | 11 frameworks | 11+ frameworks (v2.0) |
| Instalacion | npm install -g | dotnet tool install / binary |
